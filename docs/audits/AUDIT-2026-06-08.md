# Iconic CAD Technical Audit - 2026-06-08

Independent adversarial audit of the browser CAD source, with emphasis on fabrication correctness across 3D preview, BOM, fab drawings, and FreeCAD export.

## Executive Summary

The codebase has a coherent document model and several useful parity tests, but the fabrication output is not yet safe to treat as build-ready. The most serious issues are not cosmetic: top plates are under-modeled, printed cut lists round away fractional inches, second-story exports omit the floor system and can leave story gaps, the FPSF "skirt depth" field is saved and estimated in the BOM but ignored by exported geometry, and foundation quantities never enter the displayed dollar estimate.

The central "one source of truth" claim is only partly true. `enumerateMembers()` is the shared source for web 3D boxes, cut-list/BOM display, fab cards, and summary sheets. Browser FreeCAD wall solids, however, are pre-baked BREP assets loaded by module id and direction, not solids derived from `enumerateMembers()` at export time. I verified those committed BREP assets currently match their generator, so this is an architectural divergence risk rather than a currently stale asset defect.

## Phase 0 Map

Entry points:

- Browser entry: `web/index.html` loads `web/js/main.js`; `main.js` initializes 3D, UI, options, foundation modal, trade flow, home view, canvas sizing, and pricing (`web/js/main.js:1`).
- Document model: singleton `doc` in `web/js/state.js`, with `project`, `levels`, `layers`, and `entities`. Entities are walls/interior walls/foundation params (`web/js/state.js:1`).
- Model-change flow: `markModelChanged()` invalidates region cache, updates BOM, rebuilds 3D, redraws 2D, and emits `iconic:model` for trade gates (`web/js/app.js:8`).

Canonical framing representation:

- `enumerateMembers(mod)` in `web/js/members.js` is the member-list function (`web/js/members.js:40`).
- Consumers confirmed:
  - 3D preview: `buildWall3D()` loops over `enumerateMembers(mod)` (`web/js/render3d.js:199`).
  - BOM/cut list: `lineItemsForEntity()` and `cutListGrouped()` loop over `enumerateMembers(mod)` (`web/js/bom.js:50`, `web/js/bom.js:115`).
  - Fab cards: `cardSVG()` calls `enumerateMembers(mod)` and prints `cutListGrouped([entity])` (`web/js/render_fab.js:120`, `web/js/render_fab.js:280`).
  - Build summary: uses `enumerateMembers()` and `cutListGrouped()` (`web/js/render_summary.js:15`, `web/js/render_summary.js:198`).
  - FreeCAD browser export: imports `enumerateMembers()` only for height calculation; wall geometry is BREP asset translation (`web/js/fcstd.js:15`, `web/js/fcstd.js:345`).

Trade flow:

- Actual order is `['framing', 'foundation', '3d']`, not Foundation -> Framing -> 3D (`web/js/trades.js:25`).
- Framing gate is `regionForLevel(level).isEnclosed` for every required story (`web/js/trades.js:36`).
- Foundation gate is the presence of any `kind === 'foundation'` entity (`web/js/trades.js:46`).
- NEXT TRADE advances `ui.reachedTrade`; locked trades can be viewed and edited/regenerated with downstream invalidation (`web/js/trades.js:159`, `web/js/trades.js:181`).
- The options modal writes only `doc.project = { name, stories, climate }` and does not create levels, wall-height settings, geometry, or trades (`web/js/options.js:94`).

## Findings

### CAD-AUD-001 - Critical - Confirmed

**Location:** `docs/aperture_framing_reference.md:36`, `docs/aperture_framing_reference.md:102`, `web/js/members.js:63`, `web/js/members.js:107`, `generate_wall_library.py:234`

**Title:** Single top plate is emitted where the project spec requires a double top plate.

**What's wrong:** The project reference says to use a double top plate, including for exterior opening panels. The canonical enumerator emits only one `top_plate`, and the FreeCAD library generator explicitly bakes a single top plate. The enumerator comments acknowledge the mismatch rather than fixing it.

**Builder/output impact:** Every output undercounts top plates by one per panel. Studs, kings, jacks, and cripples are modeled/cut against a single top plate, so when the intended double top plate is restored, many vertical cut lengths change by 1.5 in. This affects 3D, BOM, fab cards, build summary, and FreeCAD exports.

**Trigger:** Any wall, window, door, or interior panel.

**Fix direction:** Make double top plate part of the canonical member schema. Add the second top plate explicitly, lower full-height vertical members accordingly, update generator/BREP assets, and add acceptance tests that assert both plate count and vertical member lengths.

### CAD-AUD-002 - Critical - Confirmed

**Location:** `web/js/bom.js:106`, `web/js/bom.js:118`, `web/js/render_fab.js:280`, `web/js/render_fab.js:284`, `web/js/render_summary.js:198`, `web/js/render_summary.js:235`

**Title:** Cut lists round all lengths to whole inches, producing wrong cut lengths.

**What's wrong:** `cutListGrouped()` converts every member length with `Math.round(mm / IN_TO_MM)`. Fab cards and the build-summary cut sheet print those rounded strings. The underlying enumerator has fractional cut lengths, but the fabrication output loses them.

**Builder/output impact:** Printed cut lengths are wrong. Examples from the current enumerator:

- `door_4x8_2x6_38x83`: jack studs are 81.5 in but print as 82 in; top cripples are 4.25 in but print as 4 in.
- `window_4x8_2x6_36x48`: top cripples are 15.25 in but print as 15 in.
- `window_4x10_2x6_36x48`: top cripples are 39.25 in but print as 39 in.

**Trigger:** Any module with fractional member lengths, especially doors and windows.

**Fix direction:** Replace whole-inch rounding with the existing fractional formatter, at least to nearest 1/8 in or 1/16 in, and group by exact normalized fractional length. Add tests that assert printed cut-list labels for 4.25, 15.25, 39.25, 70.5, and 81.5 in members.

### CAD-AUD-003 - Critical - Confirmed

**Location:** `web/js/state.js:55`, `web/js/state.js:69`, `web/js/render3d.js:269`, `web/js/render3d.js:274`, `web/js/render3d.js:282`, `web/js/fcstd.js:300`, `compile_from_json.py:447`

**Title:** Level 2 ignores floor-to-floor height and exports directly at tallest L1 wall top.

**What's wrong:** `ensureLevel2()` stores `L2.z_mm = FLOOR_TO_FLOOR_MM`, documented as including wall, double top plate, bearing gap, 2x12 joist depth, and subfloor. Both 3D and browser FreeCAD export ignore that `z_mm`; they compute `l2BaseZ` as `max(panelHeightMM(L1 walls))`. The terminal compiler repeats that behavior and explicitly refuses to read `levels[].z_mm`.

**Builder/output impact:** Second-story models omit the floor/rim/joist/subfloor system and place L2 framing directly on the L1 wall top. If L1 mixes 8 ft walls with 9 ft or 10 ft aperture panels, all L2 walls sit at the tallest panel height, leaving exported gaps above shorter L1 walls. A builder acting on the FreeCAD model sees a coherent-looking but structurally impossible stack.

**Trigger:** Any two-story project. Mixed-height L1 panels make the gap obvious; all-8 ft L1 panels still omit the floor system.

**Fix direction:** Decide one canonical story datum. Either use `levels[].z_mm` and emit the floor/rim/subfloor system, or block second-story export until that system exists. Add regression tests with mixed L1 heights and assert no unsupported L2 gaps.

### CAD-AUD-004 - Critical - Confirmed

**Location:** `web/js/foundation.js:52`, `web/js/foundation.js:88`, `web/js/foundation_geom.js:162`, `web/js/foundation_geom.js:166`, `web/js/foundation_geom.js:240`, `foundation_lib.py:316`, `web/js/bom.js:146`

**Title:** FPSF skirt depth is captured and estimated but ignored by generated geometry.

**What's wrong:** The foundation modal autogenerates `skirt_depth_mm` from climate frost depth and saves it on the foundation entity. `foundationSolids()` then marks `skirt_depth_mm` as reserved and never consumes it. The geometry uses only a 1 in offset plus a 2 in skirt thickness around the beam. Meanwhile, the BOM estimates EPS as `perim * p.skirt_depth_mm`.

**Builder/output impact:** 3D/FreeCAD show one foundation geometry, while the BOM estimates a different EPS area. More seriously, the exported FPSF does not include the climate-driven horizontal frost skirt/wing implied by the UI. Changing climate zone or editing skirt depth changes the BOM but not the exported geometry.

**Trigger:** Generate any foundation, especially with zones 3-8 or manual skirt-depth edits.

**Fix direction:** Implement the horizontal FPSF wing using `skirt_depth_mm`, or rename/remove the field until it is supported. Make the BOM estimate derive from the same `foundationSolids()` pieces, not a separate perimeter formula.

### CAD-AUD-005 - Critical - Confirmed

**Location:** `docs/aperture_framing_reference.md:102`, `web/js/members.js:138`, `web/js/members.js:141`, `web/js/bom.js:118`, `web/js/render_fab.js:280`

**Title:** Opening panels list OSB as strips around openings despite the repo reference requiring full sheathing.

**What's wrong:** The project reference says exterior opening panels use OSB over the whole panel face, with openings cut on install. The enumerator splits sheathing into side/top/bottom strips around the rough opening, and the fab card prints those sheathing rectangles in the cut list.

**Builder/output impact:** A builder could cut sheathing into narrow strips instead of applying full sheet sheathing, undermining shear/sheathing intent and diverging from the stated reference. The material BOM still adds whole-sheet counts, so the pick list and fab cut list disagree.

**Trigger:** Any exterior window or door module.

**Fix direction:** Represent sheathing as full sheets/panels with opening annotations if needed. Keep cut-list material counts and fab drawing instructions aligned: full sheet takeoff, field-cut opening, no strip cut list unless engineered.

### CAD-AUD-006 - High - Confirmed

**Location:** `web/js/bom.js:29`, `web/js/bom.js:35`, `web/js/bom.js:36`, `web/pricing.json:31`, `web/pricing.json:66`, `web/pricing.json:74`

**Title:** Stock BOM maps long studs to 8 ft stock and prices headers at zero.

**What's wrong:** `stockKeyFor()` returns `2x6_8ft` for every non-header 2x6 stud-like member, regardless of actual member length. The pricing catalog knows some modules need 8.5, 9, or 10 ft studs, but that data is not used. Header members return material keys like `2x8` and `2x12`, but the catalog lacks those keys, so they contribute no cost.

**Builder/output impact:** The BOM/cost estimator underbuys and underprices long panels and all structural headers. Example: a `window_4x10_2x6_36x48` king stud is 117 in, but the stock BOM counts it as an 8 ft 2x6. Header quantities appear in the cut list, but not in cost.

**Trigger:** Any 8.5 ft, 9 ft, or 10 ft panel; any window/door/garage/sliding/double-door header.

**Fix direction:** Select stock by `m.length_mm`, nominal, and role. Add priced `2x8`, `2x10`, and `2x12` stock keys. Add tests comparing aggregate stock keys against enumerated member lengths.

### CAD-AUD-007 - High - Confirmed

**Location:** `web/js/io.js:66`, `web/js/io.js:80`, `web/js/io.js:83`, `web/js/io.js:87`, `web/js/io.js:111`, `web/js/ui.js:367`, `web/js/ui.js:370`

**Title:** Loading does not reset active level or stale level stack, corrupting legacy loads after L2 work.

**What's wrong:** `loadLayout()` replaces `doc.project` and `doc.entities`, but it does not reset `doc.activeLevel`. If a loaded file lacks `levels`, it also does not reset `doc.levels` to the default single-story level. Legacy entities without `m.level` are assigned `m.level || doc.activeLevel`, so a legacy single-story file loaded while the app is on L2 can silently load its walls onto hidden L2.

**Builder/output impact:** Users can load a valid single-story file into an invisible or stale level state. Subsequent placement, trade gating, BOM grouping, and export can operate on the wrong story without an obvious error.

**Trigger:** Work on a two-story project, switch to Level 2, then load an older/legacy layout with no `levels` or per-entity `level`.

**Fix direction:** At the start of every load, reset `doc.levels`, `doc.activeLevel`, `doc.activeLayer`, trade frontier, history/future, and level defaults before applying file data. Default missing entity levels to `L1`, not current UI state.

### CAD-AUD-013 - High - Confirmed

**Location:** `web/js/bom.js:42`, `web/js/bom.js:44`, `web/js/bom.js:85`, `web/js/bom.js:160`, `web/js/bom.js:172`, `web/pricing.json:1`

**Title:** Foundation quantities are displayed but excluded from the dollar estimate.

**What's wrong:** `lineItemsForEntity()` returns line items only for `wall` and `iwall`, so foundation entities never enter `aggregateLineItems()`. `updateBOM()` computes `totalCost` only from that aggregate and the pricing catalog. Foundation quantities are computed later by `foundationEstimate()` and displayed as concrete/EPS rows, but there are no concrete/EPS catalog keys and no foundation line items contributing to `totalCost`.

**Builder/output impact:** After generating a foundation, the BOM can show concrete and EPS quantities while the "Est. cost" remains framing/hardware only. This materially understates project cost and makes the estimate internally inconsistent.

**Trigger:** Any project with a generated foundation.

**Fix direction:** Add foundation material keys to `pricing.json`, emit foundation line items from the foundation entity or from `foundationSolids()`, and compute foundation cost from the same geometry/quantity source used for the visible rows.

### CAD-AUD-014 - High - Confirmed

**Location:** `web/js/render2d.js:16`, `web/js/render2d.js:17`, `web/js/main.js:27`, `web/js/ui.js:436`, `web/js/ui.js:442`, `web/js/trades.js:82`, `web/js/trades.js:86`

**Title:** Returning from Foundation/3D to Framing can leave the framing review blank.

**What's wrong:** Foundation and 3D trades hide `#canvas-wrap` through `switchTab('3d')`. The global window resize handler still calls `resizeCanvas()`, which sets `canvas.width = wrap.clientWidth` and `canvas.height = wrap.clientHeight`; while the wrapper is hidden, those dimensions can be zero. Returning to Framing calls `switchTab('2d')`, but that only restores display state and does not call `resizeCanvas()` or `draw2d()`.

**Builder/output impact:** The document model is still intact, so `NEXT TRADE` can remain enabled and foundation regeneration still works, but the Framing trade can show a blank/nonstandard view with no visible wall modules. This creates an invalid review state during normal trade navigation.

**Trigger:** Enter Foundation/3D, let the hidden 2D canvas be resized while hidden (for example by viewport/layout resize), then click the Framing trade rail button. The model remains present but the 2D canvas has no drawable area until another resize/redraw path restores it.

**Fix direction:** When switching to `2d`, call `resizeCanvas()` after `#canvas-wrap` is visible, or make `resizeCanvas()` no-op while hidden and force a resize on return. Add a UI regression test that switches Foundation -> Framing after a resize and asserts the canvas has nonzero dimensions and visible modules.

### CAD-AUD-008 - Medium - Confirmed

**Location:** `web/js/ui.js:705`, `web/js/ui.js:709`, `web/js/ui.js:711`, `web/js/fcstd.js:295`, `web/js/fcstd.js:298`, `web/js/render_fab.js:404`, `web/js/render_summary.js:271`

**Title:** Export actions bypass trade gates and can emit incomplete models.

**What's wrong:** The export modal is always wired directly to JSON, FCStd, fab drawings, and build summary exports. The exporters only check for at least one wall/panel; they do not check `framingDone()`, `foundationDone()`, or `ui.reachedTrade`.

**Builder/output impact:** A user can export a `house.FCStd` before the framing shell is closed or before a foundation exists. That may be acceptable for a framing-only fab pack, but the UI labels it as a house export and does not warn that downstream trades are missing.

**Trigger:** Place one or more panels, do not advance through Foundation, then use the export menu.

**Fix direction:** Gate or label exports by scope. For full-house FCStd, require `ui.reachedTrade >= 2` and `foundationDone()`. For framing-only fab drawings, explicitly label the output as framing-only and include a validation warning if the shell is not closed.

### CAD-AUD-009 - Medium - Confirmed

**Location:** `web/js/members.js:4`, `web/js/fcstd.js:303`, `web/js/fcstd.js:345`, `build_lib.py:76`, `scripts/bake_geometry.py:6`

**Title:** FreeCAD wall export is not generated from `enumerateMembers()`.

**What's wrong:** `members.js` claims the FreeCAD export consumes `enumerateMembers()`. Browser FCStd export actually translates pre-baked `assets/lib/<module>__<dir>.brp` wall solids. Those solids are generated by `generate_wall_library.py` through `scripts/bake_geometry.py`, not by the JavaScript member list.

**Builder/output impact:** The one-source-of-truth claim is false for FreeCAD wall solids. If a future framing change updates `enumerateMembers()` but does not regenerate BREP assets, web 3D/fab/BOM and FreeCAD export can diverge silently.

**Trigger:** Any future change to framing math or member representation without a full `build_lib.py --verify` gate.

**Fix direction:** Either generate FreeCAD solids from the canonical member list at export time, or make the BREP generator consume the same member spec and enforce `build_lib.py --verify --no-thumbs` in CI for every framing change.

**Verification note:** I ran `python build_lib.py --verify --no-thumbs`; committed `web/assets/lib` geometry passed verification against the current generator.

### CAD-AUD-010 - Medium - Confirmed

**Location:** `docs/aperture_framing_reference.md:79`, `docs/aperture_framing_reference.md:81`, `web/js/constants.js:45`, `web/js/members.js:110`, `web/js/members.js:113`

**Title:** Interior door framing conflicts with the measured interior-door reference.

**What's wrong:** The measured interior door reference says the interior door has no separate jack studs: inner full-height studs frame the opening, with a flat 2x4 header nailed across. The shared aperture enumerator uses the exterior opening pattern for all apertures, including `idoor_4x8_2x4_38x83`, emitting jack studs from bottom plate to rough-opening top.

**Builder/output impact:** Interior door fab cards and BOM do not match the documented measured interior-door source. The output may overstate short jack pieces and understate full-height opening-side studs for interior doors.

**Trigger:** Place/export `idoor_4x8_2x4_38x83`.

**Fix direction:** Split interior and exterior aperture framing rules, or encode a framing style flag on aperture modules. Add an acceptance test using the exact measured interior-door table.

### CAD-AUD-011 - Medium - Confirmed

**Location:** `web/js/options.js:4`, `web/js/options.js:57`, `web/js/options.js:94`, `web/index.html:876`, `web/index.html:881`, `web/index.html:896`

**Title:** Per-story wall height is advertised in code comments but has no UI or model field.

**What's wrong:** The options modal header comment says it captures "per-story wall height"; the actual form state contains only name, stories, and climate zone, and the HTML only renders those fields. `doc.project` receives no wall-height settings.

**Builder/output impact:** There is no NaN propagation from missing wall-height inputs, but the subsystem is not implemented. Wall heights are still selected indirectly by module choice, and no per-story height constraint prevents mixed-height first stories under a second story.

**Trigger:** Start any project from the options modal.

**Fix direction:** Remove the claim until implemented, or add explicit per-story wall-height fields that constrain module choices and drive level heights/export.

### CAD-AUD-012 - Low - Confirmed

**Location:** `web/js/fcstd.js:423`

**Title:** Browser FCStd export depends on a CDN JSZip import.

**What's wrong:** FCStd export dynamically imports JSZip from `https://cdn.jsdelivr.net`. A local/static deployment without network access cannot export FCStd even if all project assets are present.

**Builder/output impact:** Export can fail in offline shop/site use. This is not a geometry error, but it is a reliability issue for a browser-based CAD tool.

**Trigger:** Use FCStd export offline or with CDN blocked.

**Fix direction:** Vendor JSZip locally or bundle it into the app with the rest of the static assets.
